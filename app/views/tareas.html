and explain what "Microservices Architecture" means 20 points.

A microservices architecture is a way to design software, basically it involves making the multiple parts of an app into modular pieces of software (I think modular might be the best word to describe microservices), this allows for multiple things, but mainly so that all these pieces or microservices can be used independently and upgraded independently and not the whole system. This can also extend to having multiple databases for multiple services, sometimes just copies of it, so they can all act independently. An important part of this is the communication between microservices, since microservices are to act in an independent way, their communication works in more of a RESTish way, doing requests and receiving responses from other microservices.

3. Create a grunt task to deploy your website to your own bucket in S3. 30 points.

Assuming we already have a gruntfile.js and a package.json from scaffolding, we will add the aws module from the terminal with the following command inside the project folder:

npm i grunt-aws-s3 --save-dev

We can make sure it has been installed by checking that the module shows up in the package.json

Now inside the gruntfile.js, we can add the aws settings inside the initconfig section:


aws_s3: {

options: {

accessKeyId: 'key',

secretAccessKey: 'key'

},

dist: {

options: {

bucket: successfulstudent'

},

files: [

{

expand: true,

cwd: 'dist/',

src: [ '**' ],

dest: '/'

}

]

}

}

Now to create the task and to give it a name we create the grunt task

grunt.registerTask( 'goaway', 'aws_s3:dist' );

Now we can just grunt build the project and the grunt goaway task should be available to use and deploy my website.

BUT, since we want our keys to be read only for the upload, we can place them outside the gruntfile.

Simply create a json file with content such as:

{

"AWSAccessKeyId": "key",

"AWSSecretKey": "key"

}

And now to read it inside the gruntfile we can add in the initconfig:

aws: grunt.file.readJSON( 'filewithkeys.json' ),

So we can now in the aws_s3 settings go like this:

options: {

accessKeyId: '<%= aws.AWSAccessKeyId %>',

secretAccessKey: '<%= aws.AWSSecretKey %>'

},

And done, just remember to not upload to github the file with the keys!

4. Explain how routes work in angular and a code example. 10 points.

Routes in angular, specifically while using the ngRoute module are what allows an application to become a single page application despite having multiple views.

To use it, all you have to do is add the ngRoute module inside the app.js and use the $routeProvider to specify which route goes to what view in your project, as wells as a possible controller if being used.

angular

.module('succesfulstudentApp', [

'ngRoute'

])

.config(function ($routeProvider) {

$routeProvider

.when('/about', {

templateUrl: 'views/about.html',

controller: 'AboutCtrl',

controllerAs: 'about'

})

.otherwise({

redirectTo: '/'

});

Where the when specifies which route is being used and redirects to the templateUrl, also specifies the controller and the file name where the controller is.

5. Explain how you can have more than one access key in your computer and how to

use each one. Provide code or configuration examples. 10 points.

So first we need to add extra profiles in our credentials file, such as:

[default]

aws_access_key_id = ACCESS_KEY

aws_secret_access_key = SECRET_KEY

[NotAdrian]

aws_access_key_id = ACCESS_KEY

aws_secret_access_key = SECRET_KEY

This way now if we use the aws s3 sync . s3://something  we can add - -profile NotAdrian we can use the other profile instead of default.

Example: aws s3 sync - -profile NotAdrian . s3://something
